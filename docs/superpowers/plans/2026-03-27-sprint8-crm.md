## Task 1 — Page CRM liste clients `/dashboard/crm`

### Fichiers
- `app/dashboard/crm/page.tsx` (nouveau)
- `app/dashboard/crm/layout.tsx` (nouveau, optionnel si partagé)
- `components/crm/CustomerTable.tsx` (nouveau)
- `components/crm/CrmStatCards.tsx` (nouveau)

### Étapes

1. **Créer `app/dashboard/crm/page.tsx`**
   - Server component qui fetch `/api/customers?page=1&limit=50` et les stats agrégées.
   - Passe les données à `<CustomerTable>` et `<CrmStatCards>`.

2. **`CrmStatCards`** — 4 cartes : clients inscrits, membres Gold, points distribués ce mois, récompenses utilisées.
   - Données issues d'un endpoint `/api/crm/stats` (GET, à créer) ou calculées côté server component depuis Supabase directement.

3. **`CustomerTable`** — table avec colonnes : Client (avatar + nom + email), Statut (badge), Points (barre), CA total, Dernière visite, Boisson favorite.
   - Search : input côté client, debounce 300ms, appelle `/api/customers/search?q=`.
   - Filtre statut : `<select>` → filtre local sur le tableau déjà chargé.
   - Clic sur ligne ou bouton "Voir" → `router.push('/dashboard/crm/' + id)`.

4. **Bouton "Programme"** dans la topbar → `router.push('/dashboard/crm/programme')`.

### API à créer
```
GET /api/crm/stats
→ { totalCustomers, goldCount, ptsDistributedThisMonth, rewardsUsedThisMonth }
```

### Test
- Charger la page, vérifier que la table affiche les clients de la DB.
- Taper dans le champ recherche, vérifier le filtrage.
- Cliquer "Voir" sur un client, vérifier la navigation vers la fiche.

## Task 2 — Fiche client `/dashboard/crm/[id]`

### Fichiers
- `app/dashboard/crm/[id]/page.tsx` (nouveau)
- `components/crm/CustomerProfile.tsx`
- `components/crm/CustomerOrderHistory.tsx`
- `components/crm/CustomerNotes.tsx`
- `components/crm/CustomerLoyaltyPanel.tsx`

### Étapes

1. **`app/dashboard/crm/[id]/page.tsx`** — server component.
   - Fetch en parallèle : `GET /api/customers/[id]`, `GET /api/customers/[id]/rewards`, `GET /api/customers/[id]/orders?limit=20`.
   - Layout deux colonnes (CSS grid 60/40).

2. **`CustomerProfile`** — header avec avatar (initiales calculées), nom, badge statut, téléphone, email, date d'inscription. Stat row : CA total, nb visites, ticket moyen, points actuels.

3. **`CustomerOrderHistory`** — table des 20 dernières commandes : Date, Produits (items du JSON), Montant, Points gagnés. Badge violet pour les points. Badge ambre pour les lignes "Récompense utilisée".

4. **`CustomerNotes`** — `<textarea>` avec le champ `notes` du customer. Bouton "Sauvegarder" → `PATCH /api/customers/[id]` avec `{ notes }`. Toast de confirmation.

5. **`CustomerLoyaltyPanel`** — panneau droit :
   - Barre de progression vers le niveau suivant (calcul : seuil_prochain - pts_actuels).
   - Liste récompenses disponibles / verrouillées depuis `/api/customers/[id]/rewards`.
   - Bouton "Appliquer" sur récompenses disponibles → `POST /api/loyalty/apply-reward` avec `{ customerId, rewardId }`. Puis revalidation des données.
   - Timeline des 10 dernières `loyalty_transactions`.
   - Boutons "Envoyer QR" (SMS / Email) → `POST /api/customers/[id]/send-qr` avec `{ channel: 'sms' | 'email' }`.

6. **`GET /api/customers/[id]/orders`** — à créer si absent. Requête Supabase `orders` filtrée par `customer_id`, jointure `order_items` pour les produits.

7. **`POST /api/customers/[id]/send-qr`** — à créer. Supabase Edge Function ou appel direct à un service SMS/email (Resend pour email). Pour le sprint 8, un log console suffit si l'intégration SMS n'est pas encore configurée — ne pas bloquer la feature.

### Test
- Ouvrir la fiche d'un client Gold — vérifier barre de progression, récompenses, timeline.
- Modifier la note et sauvegarder — recharger la page, vérifier la persistance.
- Cliquer "Appliquer" sur une récompense — vérifier que les points se déduisent.

### Commit
```
feat: add customer detail page with loyalty panel and order history
```

## Task 3 — Config programme fidélité `/dashboard/crm/programme`

### Fichiers
- `app/dashboard/crm/programme/page.tsx` (nouveau)
- `components/crm/LoyaltyConfigForm.tsx`
- `components/crm/RewardRow.tsx`
- `app/api/loyalty/config/route.ts` (GET + PUT)

### Étapes

1. **`GET /api/loyalty/config`** — retourne l'enregistrement `loyalty_config` du shop courant. Si absent, retourne des valeurs par défaut.

2. **`PUT /api/loyalty/config`** — reçoit le body complet et upsert dans Supabase. Validation zod : `pts_per_euro` number, `signup_bonus` number, niveaux array, récompenses array.

3. **`app/dashboard/crm/programme/page.tsx`** — server component, fetch la config, passe à `<LoyaltyConfigForm>`.

4. **`LoyaltyConfigForm`** — client component géré avec `useState` ou `react-hook-form`.
   - Toggle ON/OFF → champ `active` dans le state.
   - Section Points : 4 inputs numériques.
   - Section Niveaux : 3 cartes avec champs nom, seuil min/max, avantages textarea. Standard et Gold ont leur seuil min/max fixes (0 et ∞).
   - Section Récompenses : state local `rewards[]`. Bouton "Ajouter" push un objet vide. Bouton "Supprimer" filtre le tableau. Chaque ligne = `<RewardRow>`.
   - Bouton "Sauvegarder" → `PUT /api/loyalty/config`. Toast succès/erreur. Pas de navigation.

5. **`RewardRow`** — ligne de table avec inputs contrôlés (nom, pts, type select, valeur, niveau requis, delete).

### Test
- Modifier le nombre de points par euro, sauvegarder, recharger la page → vérifier la persistance.
- Ajouter une récompense, sauvegarder → elle apparaît dans la liste fiche client.
- Désactiver le programme avec le toggle → vérifier que le POS n'affiche plus le widget fidélité.


## Task 4 — POS loyalty widget complet

### Fichiers
- `components/pos/LoyaltyWidget.tsx` (modifier l'existant)
- `app/api/customers/search/route.ts` (vérifier / compléter)
- `app/api/loyalty/apply-reward/route.ts` (vérifier / compléter)

### Étapes

1. **Lire l'état actuel** de `LoyaltyWidget.tsx` — identifier les TODOs et le skeleton existant.

2. **État du composant** :
   ```
   phase: 'idle' | 'searching' | 'found' | 'reward_selected'
   customer: Customer | null
   selectedReward: Reward | null
   searchQuery: string
   ```

3. **Phase `idle`** — affiche un champ de recherche (téléphone ou nom) + bouton "Passer sans fidélité".

4. **Phase `searching`** — debounce 400ms sur l'input, `GET /api/customers/search?q=` → si résultat unique, passe en `found`. Si plusieurs, affiche une liste cliquable.

5. **Phase `found`** — affiche : avatar initiales, nom, badge statut, points actuels. Liste des récompenses disponibles (fetch `/api/customers/[id]/rewards`). Bouton "Continuer sans récompense". Bouton "Appliquer" sur chaque récompense disponible → passe en `reward_selected`.

6. **Phase `reward_selected`** — affiche un résumé : "Récompense : Café offert" + bandeau vert. Le POS peut finaliser la commande. À la soumission de la commande, le parent passe `customerId` et `rewardId` dans le payload de la commande.

7. **Intégration commande** — dans le handler de finalisation de commande du POS, si `customerId` présent : `POST /api/loyalty/apply-reward` après la création de la commande. Les points sont crédités automatiquement (calcul : `Math.floor(total * pts_per_euro)`).

8. **Vérifier `/api/customers/search`** — doit retourner `id, name, phone, points, tier` au minimum.

9. **Vérifier `/api/loyalty/apply-reward`** — doit accepter `{ customerId, orderId, rewardId?, orderTotal }`, créer une `loyalty_transaction` de type `earn` (et `spend` si rewardId).

### Test
- Au POS, taper un numéro de téléphone → le client doit apparaître.
- Sélectionner une récompense → elle s'affiche dans le résumé.
- Finaliser la commande → vérifier dans la fiche client que les points ont été crédités.
- Cliquer "Passer sans fidélité" → la commande se finalise normalement, sans transaction de points.

### Commit
```
feat: complete POS loyalty widget with customer search and reward application
```
